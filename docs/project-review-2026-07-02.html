4.2 데이터 완결성 (High)

HIGHST-5 · 수정주가(Naver)와 원주가(KRX/KIS) 소스 혼합작업량 M

transforms/builders.py:680-701 · sources/kis/client.py:35 · sources/naver/client.py:74-86

KIS는 FID_ORG_ADJ_PRC: "1"(원주가), Naver siseJson은 분할 반영 수정주가를 반환한다. _select_preferred_price_sources가 (date, ticker)별 krx > kis > naver 우선순위로 한 행만 남기므로 소스 경계일에 가격 점프가 발생하고, ST-1의 adjustment factor가 생기면 Naver 구간은 이중 보정이 된다.

silver.prices에 소스별 가격 기준(adjusted/raw) 컬럼을 추가하고, Naver 구간을 인접 KRX/KIS 구간과의 종가 비율로 역보정(un-adjust)하거나 최소한 소스 경계일의 return_1d를 null 처리. 기존 연속성 휴리스틱(builders.py:603-641)을 소스 경계에 확장 적용 가능.
HIGHST-6 · 부분 실패가 marker/청크 존재로 영구 스킵 — 조용한 데이터 누락작업량 M

sources/naver/adapter.py:87-88, 114-139 · sources/opendart/adapter.py:234, 278-301

Naver 백필은 티커별 실패를 삼키고 성공분만 청크 파일에 쓰는데, list_pending은 청크 파일 존재만 보고 스킵하므로 실패 티커는 재실행해도 영원히 재시도되지 않는다. DART 벌크도 동일하게 일부 실패에도 marker에 "ok"를 기록한다. OpenDART 일일 쿼터(20,000건) 초과로 배치 후반부가 전멸해도 다음 날 복구되지 않는다.

실패 존재 시 marker에 "partial\n" + 실패 목록을 기록하고 list_pending이 partial marker의 실패분만 재시도 유닛으로 생성. 최소 구현: 실패가 있으면 marker/청크를 아예 쓰지 않기(재시도 비용 < 영구 누락).
HIGHST-7 · HTTP 재시도에 429/5xx 미포함 — KIS 레이트리밋에 무방비작업량 S

http.py:32-37 (51-56, 66-71, 81-86 동일 패턴) · sources/kis/adapter.py:86-92

tenacity retry가 TimeoutException, TransportError만 재시도한다. KIS 초당 한도 초과(EGW00201)는 HTTP 500 또는 rt_cd != 0으로 도착해 즉시 실패 처리되고, adapter가 티커 단위로 삼켜 하루치 데이터가 조용히 빠진다. 고정 간격 0.05초(=20req/s)는 KIS 실전 한도와 정확히 같아 여유가 없다.

_raise_for_status에서 429/5xx를 RetryableApiError로 승격해 retry 조건에 추가, KIS msg_cd == "EGW00201"도 backoff 후 1-2회 재시도. 단 토큰 발급(kis/auth.py:67)은 "1분 1회" 정책상 재시도 제외 유지.
HIGHST-8 · KIS bronze 병합의 비결정적 dedup + 정정 데이터 폐기작업량 S

sources/kis/adapter.py:167-179

concat([existing, new]).sort([date,ticker]).unique(keep="last")에서 Polars sort는 기본 불안정 정렬이라 동일 키의 기존/신규 행 순서가 보장되지 않는다 — 어느 값이 남을지 비결정적. 또 신규 키가 0이면 쓰기를 건너뛰므로 동일 키에 값이 수정된 행(정정)은 항상 폐기된다.

병합 전 pl.lit(0/1).alias("_recency") 부여 후 sort([date,ticker,_recency])로 결정화. 변경 감지는 행수 비교 대신 anti-join으로.
HIGHST-9 · TTM 부재 — 분기 누적치가 연간 이익 자리에 들어감작업량 M

transforms/builders.py:462-474 · factors/library.py:41-54, 75-94

1단계: 팩터에서 report_type == "11011"(사업보고서)로 필터해 연간 기준 통일 + PIT 정렬 키에 is_consolidated 추가(연결 우선 결정화). 2단계: 분기 누적치 차분으로 TTM 계산(admin의 eps_ttm 로직이 이미 같은 계산을 하므로 재사용 가능).
HIGHST-11 · 사설망 토큰 우회 + CSRF 잡 실행작업량 S

admin/server.py:1681 (기본 host 0.0.0.0) · 1878-1888 (_authorized) · 1949-1956 · 1832-1846 (POST /api/jobs)

_is_local_admin_client가 loopback뿐 아니라 is_private 대역 전체를 토큰 없이 통과시키고 서버는 기본 0.0.0.0 바인딩이다. 같은 LAN의 모든 기기가 무토큰으로 잡 실행·파일 읽기가 가능하고, CORS는 요청 전송을 막지 않으므로 LAN 사용자가 방문한 임의 웹페이지가 fetch()로 CSRF 잡 실행을 유발할 수 있다. (잡 커맨드는 고정 argv라 셸 인젝션은 없음 — 무단 트리거가 문제.)

공시 스트림(dart-filings)이 이미 수집되고 있고 PIT 인프라 위에서 “공시 유형별 초과수익률(CAR) 자동 산출”이 가능한 구조다. 자사주 매입·유상증자·CB 발행·최대주주 변경 등 유형별 이벤트 스터디를 재현 가능한 파이프라인으로 만들면 그 자체가 연구 플랫폼이 된다. report_nm 규칙 분류로 시작해 Claude API로 공시 본문 요약·정밀 분류(호재/악재/중립, 규모)를 붙이면 규칙으로 안 잡히는 이벤트까지 커버.
단계
① silver.filings 빌더 구현(현재 영구 빈 뷰 — ST-21과 연계) → ② 유형 분류기(규칙 기반) → ③ 이벤트 스터디 러너(이벤트일 ±N일 CAR, 시장 조정 수익률) → ④ LLM 분류 파이프라인(배치, 일 수백 건 규모라 비용 미미) → ⑤ 이벤트 팩터(예: 자사주 매입 공시 후 보유).
LT-4 · 멀티팩터 리서치 인프라: 팩터 3종 → 리서치 플랫폼플랫폼 완성도
왜 가치가 큰가
현재는 팩터 인벤토리(3종)와 인프라(크로스섹션 유틸 전무, 엔진이 Factor.rebalance도 무시) 모두 초기 단계라, 인프라를 한 번 깔면 팩터 추가가 “정의 20줄”짜리 작업이 된다.
단계
① 크로스섹션 유틸(z-score, winsorize, rank, 섹터·사이즈 중립화) + 팩터 방향 메타데이터 → ② 신규 팩터: book-to-price·배당수익률(silver.dividends 이미 존재)·GP/A·발생액·저변동성·1M 반전·52주 신고가 근접도 → ③ 멀티팩터 결합기(동일가중 z-score부터) → ④ 팩터별 자동 tear sheet(IC 시계열, 분위수 수익률, 턴오버, 최대낙폭)를 기존 리포트 파이프라인(HTML)에 통합 → ⑤ 리밸런스 주기 파라미터화.
LT-5 · 백테스트 → 실행 브릿지: KIS 모의투자 페이퍼 트레이딩차별화 기능
왜 독창적인가
KIS 인증·토큰 캐시 인프라가 이미 있으므로, 백테스트에서 검증된 전략을 KIS 모의투자 계좌에 자동 리밸런싱하는 루프까지의 거리가 짧다. “백테스트 플랫폼”이 “전략 운용 플랫폼”이 되는 단계 — 개인 프로젝트로서는 흔치 않은 완결성이며, 실체결 데이터로 백테스트 비용 모델(ST-21의 거래세 건)을 역검증하는 선순환도 생긴다.
단계
① 월간 리밸런싱 목표 포트폴리오 → 주문 시트 생성기(현 보유와의 diff) → ② KIS 모의투자 주문 API 어댑터 → ③ 체결·슬리피지 기록을 gold에 적재 → ④ 실전 계좌는 주문 시트 수동 승인 모드로만(안전장치). systemd 타이머 기반 월간 자동화.
LT-6 · pre-2010 백필 완성: 30년 장기 백테스트데이터 해자
왜 가치가 큰가
sources/pre2010은 스키마만 있고 미구현(커버리지 0%)이다. 하지만 docs/kospi-top-removal-impact-1996-2025.md 연구가 이미 1996년부터의 분석 수요를 증명했고, 아키텍처 문서 로드맵 9단계에도 계획되어 있다. 한국 시장의 1996~2010 구간(IMF·닷컴·카드사태)을 포함한 PIT 백테스트가 가능해지면 팩터 검증의 통계적 신뢰도가 질적으로 달라진다.
단계
① 커버리지 리포트(어느 소스가 어느 구간을 주는지) → ② 상폐 종목 포함 여부 검증(생존편향 — ST-4와 직결) → ③ identity 연속성(티커 재사용이 이 구간에서 특히 심함 — ST-19의 감지 로직이 선행 조건) → ④ 연도별 백필 파이프라인 재사용.
LT-7 · finance-pi MCP 서버: 자연어 리서치 코파일럿독창성 최고
왜 독창적인가
DuckDB 카탈로그(analytics.* 뷰)와 PIT API가 이미 구조화되어 있으므로, 이를 MCP(Model Context Protocol) 서버로 노출하면 Claude 같은 LLM이 “2015년 이후 PBR 0.5 미만에서 1년 보유 수익률 분포 보여줘” 수준의 질의를 카탈로그 스키마를 보고 직접 SQL로 변환·실행하게 할 수 있다. 개인 데이터 레이크가 대화형 리서치 어시스턴트가 되는 구성으로, 국내 주식 PIT 데이터에 대해서는 전례가 거의 없다.
단계
① 읽기 전용 DuckDB 커넥션 + 스키마 카탈로그를 도구로 노출하는 MCP 서버(Python SDK, 소규모) → ② 안전장치: read-only·행수 상한·타임아웃 → ③ 백테스트 실행 도구(팩터명·기간 파라미터) 추가 → ④ 리포트 생성 도구(기존 HTML 리포트 파이프라인 재사용). admin의 /api/docs 스키마 문서가 도구 설명으로 재사용 가능.
LT-8 · 데이터 신뢰성 등급과 능동 알림: “조용한 실패”의 구조적 종식운영 완성도
왜 필요한가
이번 검토에서 발견된 문제의 상당수(부분 실패 영구 누락, corporate_actions 부재, 빈 뷰)는 “실패가 조용하다”는 공통 원인을 가진다. 소스 간 cross-check(KIS vs Naver 종가 대사)와 데이터셋별 신선도·완결성 등급을 상시 계산하고, 임계 위반 시 텔레그램/디스코드로 능동 통지하면 같은 부류의 문제가 재발해도 하루 안에 드러난다.
단계
① 데이터셋별 SLA 정의(최신 파티션 나이, 행수 추세, 소스 대사 오차) → ② 기존 DQ 리포트를 일일 등급 테이블(gold.dq_scorecard)로 물질화 → ③ daily 잡 말미에 위반 시 웹훅 통지 → ④ admin 대시보드에 신뢰도 등급 배지 표시(스크리너 API 응답에도 데이터 신선도 메타 포함).
